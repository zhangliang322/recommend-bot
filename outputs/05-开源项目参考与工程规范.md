# 开源项目参考与工程规范

## 1. 原则

本项目尽量采用成熟开源项目和稳定库进行模块化拼接，不从零开发通用基础设施。

做法：

- 机器人框架优先参考 AstrBot。
- 微信适配保留 Wechaty 作为备选。
- Python 事件和插件模型可参考 NoneBot2。
- 定时、ORM、数据校验、测试、图片渲染等使用成熟库。
- 业务核心只开发本项目独有部分：选品评分、推荐池、一图流卡片、私域 / 公域内容策略、运营反馈闭环。
- TikTok、Instagram、小红书和电商平台数据采集优先接入现成 GitHub 项目、官方 API、联盟 API 或第三方 API，不从零开发采集器。

## 4. 数据源复用规范

所有采集能力必须通过统一适配器接入：

```text
CollectorAdapter
  collect_keywords()
  collect_product()
  normalize()
  health_check()
```

平台策略：

| 平台 | 优先方案 |
| --- | --- |
| TikTok | TikTok-Api 或 TikHub / EnsembleData / Apify 等第三方 API |
| Instagram | Instaloader 或 Apify / SerpApi |
| 小红书 | MediaCrawler，必要时保留人工导入 |
| Amazon / Google Shopping | 官方 API 或 SerpApi Shopping |
| 1688 / 淘宝 / 拼多多 | 官方开放平台 / 联盟 API |
| AliExpress | Affiliate API 或成熟 SDK |
| Temu / SHEIN | 第三方 API、榜单源或人工导入 |

合规要求：

- 只采集公开信息或授权接口返回的数据。
- 不采集私信、隐私资料、非公开内容。
- 不绕过付费、权限、登录限制。
- 对需要登录态的采集器，账号、cookie、token 必须放在环境变量或密钥管理中。
- 对所有采集器设置限流、重试、健康检查和降级。
- 保留 `raw_json`，但不得长期保存不必要的个人隐私字段。

## 5. 模块边界

```text
机器人层
  只处理消息、指令、权限、群配置、发送图片和链接。

业务服务层
  处理数据导入、商品标准化、评分、推荐池、文案和卡片生成。

数据访问层
  只处理数据库读写，不写业务判断。

渲染层
  只处理一图流卡片生成，不参与推荐排序。

风控层
  处理白名单、黑名单、敏感词、频率限制、暂停推送。
```

## 6. 代码规范

- 使用类型注解。
- 核心对象用 Pydantic model 或 dataclass。
- 配置不得硬编码在代码中。
- 每个模块保持单一职责。
- 新增业务逻辑必须有对应测试。
- 机器人发送逻辑必须支持 dry-run 和 mock。
- 日志中不得记录客户隐私、密钥、完整 token。

## 8. 许可证约束

- AstrBot 当前仓库显示为 AGPL-3.0，适合参考架构和插件机制；直接复制代码前必须评估许可证义务。
- Wechaty 仓库显示为 Apache-2.0，仍需记录引用来源和依赖版本。
- TikTok-Api、Instaloader、MediaCrawler、Crawlee、Scrapy、SerpApi SDK 等采集相关项目必须逐一记录许可证和平台条款风险。
- 第三方依赖必须统一记录在 `THIRD_PARTY_NOTICES.md`。
- 商业化前建议做一次完整开源许可证审查。
